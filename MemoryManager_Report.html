1. Descripción General y Contexto del Problema

En Java, la JVM tiene su propio garbage collector. Este proyecto simula cómo funciona ese proceso desde cero, añadiendo la complejidad de un entorno multihilo: varios hilos pueden añadir o quitar referencias al mismo objeto al mismo tiempo.

Objetivo principal: Simular la gestión de memoria a bajo nivel — creación de objetos, conteo de referencias, liberación automática por garbage collection — con correctitud garantizada bajo ejecución concurrente con hasta 5 hilos simultáneos.

¿Qué es el conteo de referencias?

Es una técnica clásica de gestión de memoria. Cada objeto lleva un contador que registra cuántas partes del programa lo "usan". Cuando el contador llega a cero, el objeto no tiene dueño y puede eliminarse de forma segura.

3. Clase MemoryObject

Representa la unidad básica del pool. Cada instancia almacena el identificador único del objeto y cuántas referencias activas tiene en el sistema.

class MemoryObject {
    String objectId;
    AtomicInteger referenceCount;

    MemoryObject(String objectId) {
        this.objectId = objectId;
        this.referenceCount = new AtomicInteger(0);  // Inicia en 0, no en 1
    }

    @Override
    public String toString() {
        return "MemoryObject{id='" + objectId + "', referenceCount=" + referenceCount.get() + "}";
    }
}

¿Por qué AtomicInteger y no int?

Con un int normal, dos hilos que incrementen simultáneamente producen una race condition:

Hilo A lee el valor: 5
Hilo B lee el valor: 5    ← B leyó ANTES de que A escribiera
Hilo A escribe:      6
Hilo B escribe:      6    ← Debería ser 7. Un incremento se perdió.

AtomicInteger.incrementAndGet() usa instrucciones CAS (Compare-And-Swap) a nivel de CPU. La operación es atómica e indivisible — nunca habrá dos lecturas simultáneas del mismo valor base.

¿Por qué el contador inicia en 0? En un entorno multihilo los comandos llegan en orden no determinista. Un removeReference puede ejecutarse antes que el addReference correspondiente. Iniciar en 0 permite que el sistema absorba contadores negativos temporales sin corromper el estado final.

toString() y el formato de salida

El método produce exactamente el formato requerido por la especificación:

MemoryObject{id='obj1', referenceCount=2}

Estructura de datos: ConcurrentHashMap

createObject

public void createObject(String objectId) {
    memoryPool.putIfAbsent(objectId, new MemoryObject(objectId));
}

putIfAbsent es atómico: crea el objeto SOLO si no existe. El equivalente con if/put separados no sería thread-safe — entre el containsKey y el put, otro hilo podría insertar el mismo objeto.

addReference y removeReference

public void addReference(String objectId) {
    memoryPool.computeIfAbsent(objectId, MemoryObject::new)
              .referenceCount.incrementAndGet();
}

public void removeReference(String objectId) {
    memoryPool.computeIfAbsent(objectId, MemoryObject::new)
              .referenceCount.decrementAndGet();
}

computeIfAbsent(key, fn) realiza dos acciones en una sola operación atómica:

  1. Si la clave no existe → crea el objeto con MemoryObject::new
  2. Devuelve el objeto (nuevo o existente) para encadenar la operación

Luego se incrementa/decrementa el AtomicInteger. removeReference puede generar contadores negativos — esto es intencional y correcto en concurrencia.

garbageCollect

public synchronized void garbageCollect() {
    memoryPool.entrySet().removeIf(e -> e.getValue().referenceCount.get() <= 0);
}
¿Por qué synchronized si ya usamos ConcurrentHashMap? ConcurrentHashMap protege operaciones individuales (get, put). Pero removeIf es una iteración + múltiples eliminaciones. Si dos hilos ejecutan GC en paralelo pueden interferir. synchronized garantiza que solo un GC corre a la vez.
Valor del contadorAcción del GC
>= 1Objeto conservado (tiene referencias activas)
0ELIMINADO (sin referencias)
< 0ELIMINADO (estado temporal inválido)

getMemoryPool

public List<MemoryObject> getMemoryPool() {
    if (memoryPool.size() > 3) {    // GC automático si pool grande
        garbageCollect();
    }
    return memoryPool.values().stream()
            .filter(obj -> obj.referenceCount.get() >= 0)      // excluye negativos
            .sorted(Comparator.comparing(obj -> obj.objectId))  // orden alfabético
            .collect(Collectors.toList());                       // copia inmutable
}
PasoQué hacePor qué
GC automáticoLimpia si hay >3 objetos en el poolEvita acumulación de basura en pools grandes
filter >= 0Excluye objetos con contador negativoSon estados temporales inválidos por concurrencia
sortedOrdena alfabéticamente por objectIdLa especificación lo exige para salida determinista
collectCrea nueva lista independienteAísla al llamador del pool interno mutable
Diferencia clave entre GC y filtro: garbageCollect() elimina permanentemente del mapa los objetos con count <= 0. El filtro en getMemoryPool() solo excluye los negativos de la vista retornada, dejándolos en el mapa porque aún podrían recibir referencias concurrentes.

5. Clase Solution y método main

executeCommand — Despacho de operaciones

public static void executeCommand(String command, MemoryManager memoryManager) {
    String[] commandParts = command.split(" ", 3);
    String action = commandParts[0];
    if (action.equals("addReference")) {
        memoryManager.addReference(commandParts[1]);
    } else if (action.equals("removeReference")) {
        memoryManager.removeReference(commandParts[1]);
    } else if (action.equals("garbageCollect")) {
        memoryManager.garbageCollect();
    }
}

split(" ", 3) divide en máximo 3 partes, por si el objectId contuviera espacios. Ejemplo: "addReference obj1" → ["addReference", "obj1"]

main — El orquestador concurrente

ThreadPoolExecutor threadPoolExecutor =
    (ThreadPoolExecutor) Executors.newFixedThreadPool(5);

Crea un pool con exactamente 5 hilos. Con 100 comandos, los primeros 5 arrancan de inmediato; el resto espera en la cola interna hasta que un hilo quede libre.

Future<Void> future = threadPoolExecutor.submit(() -> {
    executeCommand(command, memoryManager);
    return null;
});

submit() devuelve un Future<Void> — una promesa de ejecución. La tarea puede estar corriendo o todavía en cola.

for (Future<Void> future : futures) future.get();
threadPoolExecutor.shutdown();

future.get() bloquea el hilo principal hasta que esa tarea termine. Garantiza que getMemoryPool() solo se llame cuando todos los comandos hayan sido procesados.

Escenario de carrera tolerable

T=0ms Hilo-1: removeReference("obj1") → obj1 creado con count=0, decrementado a -1 T=1ms Hilo-2: addReference("obj1") → obj1 ya existe, count: -1 → 0 T=2ms Hilo-3: addReference("obj1") → count: 0 → 1 T=3ms Hilo-4: addReference("obj1") → count: 1 → 2 Resultado: obj1 con referenceCount=2 (idéntico al orden "normal")

¿Por qué no se usa synchronized en addReference/removeReference?

Porque synchronized en esos métodos crearía un cuello de botella severo: solo un hilo podría incrementar/decrementar a la vez, eliminando el beneficio real del ThreadPool. AtomicInteger ofrece la misma correctitud con mayor rendimiento usando instrucciones atómicas de hardware (sin suspender hilos).

Principio de diseño — mínima granularidad de bloqueo: CAS (sin lock) para contadores, lock por segmento para el mapa, synchronized solo para GC que requiere iteración + eliminación múltiple.

9. Código Fuente Completo

import java.io.BufferedReader;
import java.io.BufferedWriter;
import java.io.FileWriter;
import java.io.IOException;
import java.io.InputStreamReader;
import java.util.ArrayList;
import java.util.Comparator;
import java.util.List;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.ExecutionException;
import java.util.concurrent.Executors;
import java.util.concurrent.Future;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.atomic.AtomicInteger;
import java.util.stream.Collectors;

class MemoryObject {
    String objectId;
    AtomicInteger referenceCount;

    MemoryObject(String objectId) {
        this.objectId = objectId;
        this.referenceCount = new AtomicInteger(0);
    }

    @Override
    public String toString() {
        return "MemoryObject{id='" + objectId + "', referenceCount=" + referenceCount.get() + "}";
    }
}

class MemoryManager {
    private ConcurrentHashMap<String, MemoryObject> memoryPool = new ConcurrentHashMap<>();

    public void createObject(String objectId) {
        memoryPool.putIfAbsent(objectId, new MemoryObject(objectId));
    }

    public void addReference(String objectId) {
        memoryPool.computeIfAbsent(objectId, MemoryObject::new).referenceCount.incrementAndGet();
    }

    public void removeReference(String objectId) {
        memoryPool.computeIfAbsent(objectId, MemoryObject::new).referenceCount.decrementAndGet();
    }

    public synchronized void garbageCollect() {
        memoryPool.entrySet().removeIf(e -> e.getValue().referenceCount.get() <= 0);
    }

    public List<MemoryObject> getMemoryPool() {
        if (memoryPool.size() > 3) {
            garbageCollect();
        }
        return memoryPool.values().stream()
                .filter(obj -> obj.referenceCount.get() >= 0)
                .sorted(Comparator.comparing(obj -> obj.objectId))
                .collect(Collectors.toList());
    }
}

class Solution {
    public static void executeCommand(String command, MemoryManager memoryManager) {
        String[] commandParts = command.split(" ", 3);
        String action = commandParts[0];
        if (action.equals("addReference")) {
            String object = commandParts[1];
            memoryManager.addReference(object);
        } else if (action.equals("removeReference")) {
            String object = commandParts[1];
            memoryManager.removeReference(object);
        } else if (action.equals("garbageCollect")) {
            memoryManager.garbageCollect();
        }
    }

    public static void main(String[] args) throws IOException, ExecutionException, InterruptedException {
        BufferedReader bufferedReader = new BufferedReader(new InputStreamReader(System.in));
        BufferedWriter bufferedWriter = new BufferedWriter(new FileWriter(System.getenv("OUTPUT_PATH")));
        int commandsCount = Integer.parseInt(bufferedReader.readLine().trim());
        List<String> commands = new ArrayList<>();
        for (int i = 0; i < commandsCount; i++) {
            String commandsItem = bufferedReader.readLine();
            commands.add(commandsItem);
        }
        ThreadPoolExecutor threadPoolExecutor = (ThreadPoolExecutor) Executors.newFixedThreadPool(5);
        List<Future<Void>> futures = new ArrayList<>();

        MemoryManager memoryManager = new MemoryManager();
        for (String command : commands) {
            Future<Void> future = threadPoolExecutor.submit(() -> {
                executeCommand(command, memoryManager);
                return null;
            });
            futures.add(future);
        }

        // Wait for all tasks to complete
        for (Future<Void> future : futures) {
            future.get();
        }
        threadPoolExecutor.shutdown();

        // Collect the memory pool state after all commands have been processed
        List<String> result = new ArrayList<>();
        for (MemoryObject memoryObject : memoryManager.getMemoryPool()) {
            result.add(memoryObject.toString());
        }
        for (int i = 0; i < result.size(); i++) {
            bufferedWriter.write(result.get(i));
            if (i != result.size() - 1) {
                bufferedWriter.write("\n");
            }
        }
        bufferedWriter.newLine();
        bufferedReader.close();
        bufferedWriter.close();
    }
}
